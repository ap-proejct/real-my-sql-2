## 5. INSERT
- INSERT 문장이 동시에 실행되는 경우 쿼리보다는 테이블 구조가 성능에 더 큰 영향을 미친다
- SELECT와 INSERT를 동시에 빠르게 만드는 테이블 구조는 없고 어느정도 타협을 보면서 설계해야 한다

### 1. 고급 옵션
#### 1). INSERT IGNORE
- 프라이머리 키나 유니크 인덱스 값이 이미 존재하는 경우 무시하고 다음 레코드를 처리할 수 있게 해준다
- 유니크 인덱스 뿐만 아니라 데이터 타입이 일치하지 않아서 INSERT 할 수 없는 경우에도 사용할 수 있고 이럴 경우 기본 값을 넣어준다

```sql
INSERT IGNORE INTO salaries (emp_no, salary, from_date, to_date) VALUES
    (10001, 60117, '1986-06-26', '1987-06-26'),
    (10001, 62102, '1986-06-26', '1987-06-26'),
    (10001, 66074, '1986-06-26', '1987-06-26');
```

#### 2). INSERT ...ON DUPLICATE KEY UPDATE
- 프라이머리 키나 유니크 인덱스 중복이 발생하면 UPDATE 문장의 역할을 수행하게 해준다
- REPLACE은 내부적으로는 DELETE, INSERT 조합으로 작동하지만 `DUPLICATE`문장은 UPDATE하는 방식으로 작동한다

```sql
-- 기존 레코드 값에 +1을 UPDATE 한다
INSERT INTO daily_statistic (target_date, stat_name, stat_value)
VALUES (DATE(NOW()), 'VISIT', 1)
ON DUPLICATE KEY UPDATE stat_value = stat_value + 1;

-- VALUES(stat_value)를 사용하면 stat_value 칼럼에 INSERT하려고 했던 값을 반환한다
INSERT INTO daily_statistic
    SELECT DATE(visited_at), 'VISIT', COUNT(*)
    FROM access_log
    GROUP BY DATE(visited_at)
    ON DUPLICATE KEY UPDATE stat_value = stat_value + VALUES(stat_value);

-- 8.0.20 부터는 VALUES(stat_value) Deprecated 되므로 문법을 교체하는 것이 좋다
-- INSERT되는 레코드에 대해 별칭을 부여해서 참조하면 된다
INSERT INTO daily_statistic
SELECT target_date, stat_name, stat_value
FROM (
    SELECT DATE(visited_at) target_date, 'VISIT' stat_name, COUNT(*) stat_value
    FROM access_log
    GROUP BY DATE(visited_at)
) stat
ON DUPLICATE KEY UPDATE
daily_statistic.stat_value = daily_statistic.stat_value + stat.stat_value;
```

### 2. LOAD DATA 명령 주의사항
- MySQL 엔진과 스토리지 엔진을 최소화하고 스토리지 엔진이 직접 데이터를 적재하기 때문에 매우 빠르다
- 하지만 다음과 같은 단점이 있다
  - 단일 스레드로 실행
  - 단일 트랜잭션으로 실행
- 테이블 간 데이터 복사 작업이라면 `INSERT ... SELECT ...`으로 부분적으로 잘라서 하는 것이 좋다

### 3. 성능을 위한 테이블 구조
- INSERT의 성능은 테이블 구조에 의해 많이 결정된다

#### 1). 대량 INSERT 성능
- 대량 INSERT를 한다면 프라이머리 키 기준으로 미리 정렬해서 구성하는 것이 성능에 도움이 될 수 있다
- 프라이머리 키로 정렬돼 있지 않으면 프라이머리 키를 검색해서 레코드가 저장될 위치를 찾아야 한다
- 세컨더리 인덱스도 정렬된 순서대로 INSERT될 수 있다면 더 빠른 성능을 얻을 수 있다

#### 2). 프라이머리 키 선정
- INSERT가 매우 많이 실행되는 테이블이라면 단조 증가 또는 단조 감소 패턴의 값을 선택하는 것이 좋다
- 상품이나, 사용자 정보같이 읽기 비율이 압도적으로 높은 경우에는 SELECT 쿼리를 빠르게 만드는 방향으로 설계해야 한다
  - SELECT에 최적화된 프라이머리 키는 단조 증가 또는 단조 감소 패턴과는 거리가 멀다

#### 3). Auto-Increment 칼럼
- INSERT에 최적화된 테이블을 생성하기 위해서는 두 가지 요소를 갖춰야 한다
  - 단조 증가 또는 단조 감소되는 값으로 프라이머리 키 선정
  - 세컨더리 인덱스 최소화
- InnoDB는 프라이머리 키로 클러스터링되기 때문에 자동 증가(Auto Increment) 칼럼을 이용하면 클러스터링되지 않은 테이블 효과를 얻을 수 있다
  - Auto Increment를 쓰면 항상 뒤에 insert되기 때문에 클러스터링 인덱스의 정렬 비용이 거의 없어져 마치 일반 힙 테이블(heap table)처럼 동작한다.
- 자동 증가 값의 채번을 위해 잠금이 필요한데, 이를 AUTO-INC 잠금이라고 한다
  - `innodb_autoinc_lock_mode = 0`: AUTO-INC 잠금을 걸고 한 번에 1씩만 증가된 값을 가져온다
  - `innodb_autoinc_lock_mode = 1`: 레코드 한 건씩 INSERT하는 쿼리는 잠금을 사용하지 않고 뮤텍스를 이용하고, 여러 레코드를 업로드하는 경우에는 `AUTO-INC`잠금을 걸고 필요한 만큼 값을 가져와 사용한다
  - `innodb_autoinc_lock_mode = 2`: 자동 증가 값을 미리 할당받아서 처리하는 가장 빠른 방식이다. 하지만 연속성은 보장하지 않고 쿼리 기반 복제(SBR)를 사용하는 경우 레플리카 서버의 자동 증가 값이 동기화되지 못할 수도 있다
  - 기본값은 2로 바이너리 로그 포맷이 `ROW`로 변경됐기 때문인데, `STATEMENT` 바이너리 포맷으로 사용 중이라면 1로 설정해야 한다

## 6. UPDATE와 DELETE
- 여러 테이블을 조인해서 레코드를 변경하거나 삭제하는 기능도 제공한다

### 1. UPDATE ... ORDER BY ... LIMIT n
- ORDER BY와 LIMIT를 동시에 사용해 상위 몇 건만 변경 및 삭제하는 것도 가능하다
- 바이너리 로그 포맷이 문장(STATEMENT) 기반 복제에서는 주의가 필요하다

```sql
DELETE FROM employees
ORDER BY last_name
LIMIT 10;
```

### 2. JOIN UPDATE
- 조왼되는 모든 테이블에 읽기 잠금이 걸리고, 칼럼이 변경되는 테이블은 쓰기 잠금이 걸린다
- 데드락 유발 가능성이 높아 빈번하게 사용하는 것은 피하는 것이 좋다
- 조인이 있으므로 조인 순서에 따라서 성능이 달라질 수 있다
- JOIN UPDATE에서는 GROUP BY나 ORDER BY를 사용할 수 없다

```sql
-- 원하는 조인의 방향을 알려주고 싶다면 STRAIGHT_JOIN 키워드를 사용하면 된다
-- 또는 JOIN_ORDER 옵티마이저 힌트를 사용해도 된다
UPDATE departments d,
    (SELECT de.dept_no, COUNT(*) AS emp_count
    FROM dept_emp de
    GROUP BY de.dept_no) dc
    SET d.emp_count = dc.emp_count
    WHERE dc.dept_no = d.dept_no;

UPDATE (SELECT de.dept_no, COUNT(*) AS emp_count
        FROM dept_emp de
        GROUP BY de.dept_no) dc
        STRAIGHT_JOIN daprtments d ON dc.dept_no = d.dept_no
        SET d.emp_count = dc.emp_count;

UPDATE /*+ JOIN_ORDER (dc, d) */
    (SELECT de.dept_no, COUNT(*) AS emp_count
     FROM dept_emp de
     GROUP BY de.dept_no) dc
     JOIN departments d ON dc.dept_no = d.dept_no
     SET d.emp_count = dc.emp_count
```

### 3. 여러 레코드 UPDATE
- 레코드 생성(Row Constructor)문법을 이용해 레코드별 서로 다른 값을 업데이트 할 수 있다
- `VALUES ROW(...), ROW(...)` 문법을 사용하면 임시 테이블을 생성하는 효과를 얻을 수 있다

```sql
-- 임시 테이블 new_user_level을 만들어서 조인해서 업데이트 수행하는 효과를 낼 수 있다
UPDATE user_level ul
    JOIN (VALUES ROW(1, 1),
                 ROW(2, 4)) new_user_level (user_id, user_lv)
                            ON new_user_level.user_id = ul.user_id
    SET ul.user_lv = ul.user_lv + new_user_level.user_lv;
```

### 4. JOIN DELETE
- JOIN UPDATE와 마찬가지로 STARIGHT_JOIN 이나 JOIN_ORDER로 조인 순서를 옵티마이저에게 지시할 수 있다
```sql
-- 하나의 테이블에서 레코드를 삭제하는 기본 방법
DELETE e
FROM employees e, dept_emp de, departments d
WHERE e.emp_no = de.emp_no AND de.dept_no = d.dept_no AND d.dept_no = 'd001';

-- 두 개의 테이블에서 동시에 삭제하는 방법
DELETE e, de
FROM employees e, dept_emp de, departments d
WHERE e.emp_no = de.emp_no AND de.dept_no = d.dept_no AND d.dept_no = 'd001';
```

## 7. 스키마 조작(DDL)
- 스토어드 프로시저나 함수, DB나 테이블 등을 생성하거나 변경하는 대부분의 명령이 DDL에 해당된다

### 1. 온라인 DDL
- 8.0부터 스키마 변경 작업은 MySQL 서버에 내장된 온라인 DDL 기능으로 처리가 가능해졌다

#### 1). 온라인 DDL 알고리즘
- 온라인 DDL은 스키마를 변경하는 도중에도 다른 커넥션에서 해당 테이블의 데이터를 변경하거나 조회하는 작업을 가능하게 해준다
- 온라인 DDL은 `ALGORITHM`과 `LOCK`옵션을 이용해 어떤 모드로 스키마 변경을 실행할지를 결정할 수 있다
- `old_alter_table` 시스템 변수를 이용해 ALTER TABLE 명령이 온라인 DDL으로 작동할 지 아니면 예전 방식으로 처리할지를 결정할 수 있다
  - 기본값은 OFF로 자동으로 온라인 DDL이 활성화된다
- ALTER TABLE 명령을 실행하면 아래와 같은 순서로 스키마 변경에 적합한 알고리즘을 찾는다
  - ALGORITHM=INSTANT로 스키마 변경이 가능한지 확인 후, 가능하다면 선택
  - ALGORITHM=INPLACE 스키마 변경이 가능한지 확인 후, 가능하다면 선택
  - ALGORITHM=COPY 알고리즘 선택
  - 우선순위가 낮을수록 스키마 변경을 위해 더 큰 잠금과 많은 작업을 필요로 하고 서버의 부하도 많이 발생시킨다

- INSTANT
  - 테이블 데이터는 전혀 변경하지 않고, 메타데이터만 변경하고 작업을 완료한다
  - 메타데이터만 변경하기 때문에 LOCK 옵션은 명시할 수 없다
- INPLACE
  - 임시 테이블로 데이터를 복사하지 않고 스키마 변경을 실행한다
  - 내부적으로는 테이블 리빌드를 실행할 수도 있고, 테이블의 모든 레코드를 리빌드해야 하기 때문에 크기에 따라 많은 시간이 소요될 수도 있다
- COPY
  - 변경된 스키마를 적용한 임시 테이블을 생성하고, 복사한후 RENAME해서 스키마 변경을 완료하다
  - 이 방법은 읽기만 가능하고 DML은 실행할 수 없다

### 4. 테이블 변경

#### 1). 테이블 생성
- 문자열 타입은 반드시 칼럼에 최대한 저장할 수 있는 문자 수를 명시해야 한다
- 숫자 타입은 선택적으로 길이를 가질 수 있지만, 단순히 값을 표시할 때 보여줄 길이를 지정하는 것이다

```sql
CREATE [TEMPORARY] TABLE [IF NOT EXISTS] tbl_name (
    (create_definition,...)
    [table_options]
    [partition_options]
) ENGINE=INNODB;
```

#### 2). 테이블 구조 조회
- `SHOW CREATE TABLE`명령을 사용하면 테이블의 `CREATE TABLE`문장을 표시해준다
  - 최초 테이블 생성할 때 사용자가 실행한 내용을 보여주는 것이 아니라 메타 정보를 읽어서 보여주는 것이다
  - `SHOW CREATE TABLE employees` 명령은 칼럼의 목록과 인덱스, 외래키 정보를 동시에 보여주기 때문에 테이블 구조를 확인할 때 유용하다
- `DESC employees`는 칼럼 정보를 보기 편한 표 형태로 표시해준다
  - 인덱스 칼럼의 순서나 외래키, 테이블 자체의 속성은 보여주지 않으므로 테이블의 전체적인 구조를 한 번에 확인하기는 어렵다

#### 3). 테이블 구조 변경
```sql
ALTER TABLE employees
    CONVERT TO CHARACTER SET UTF8MB4 COLLATE UTF8MB4_GENERAL_CI,
    ALGORITHM=INPLACE, LOCK=NONE;
```

#### 6). 테이블 구조 복사
```sql
-- 모든 칼럼과 인덱스가 같은 테이블을 생성한다
CREATE TABLE temp_employees LIKE employees;

-- 데이터까지 복사
INSERT INTO temp_employees SELECT * FROM employees;
```

### 칼럼 변경
```sql
-- 테이블 마지막 새로운 칼럼 추가
ALTER TABLE employees ADD COLUMN emp_telno VARCHAR(20),
    ALGORITHM=INSTANT;

-- emp_no 칼럼 다음에 칼럼 추가
ALTER TABLE employees ADD COLUMN emp_telno VARCHAR(20) AFTER emp_no,
    ALGORITHM=INSTANT;

-- 데이터 타입 변경
ALTER TABLE salaries MODIFY salary VARCHAR(20),
    ALGORITHM=COPY, LOCK=SHARED;
```

#### 4). 인덱스 가시성 변경
